## Usage

Popovers respond to a user with nice-to-know information in context. A user should get an obvious, immediate benefit from the information in the Popover—otherwise, they’ll likely ignore it.

**Use when:**

- A user’s action activates a tidbit of nice-to-know, but not essential, information.
- Highlighting up to three UI elements, such as when introducing a new feature.
- Displaying low- to medium-priority information.
- An illustration or descriptive text is helpful to clarify a UI element’s purpose.

**Do not use when:**

- Explaining a familiar, expected, or standard interaction, such as how to fill in a Form field. Instead, omit the unnecessary instructions.
- A user would benefit from persistent, visible instructions essential to completing a task. Instead, use plain UI text.
- Displaying a short, inline description of an element’s purpose. Instead, use a [Tooltip](/components/tooltip).
- A robust procedure or an image of UI is needed to explain an interaction. Instead, use a [Modal](/components/modal) or [Link](/components/link) to a help center article.

## Behavior

### Width​

- Popovers have a minimum width of 225px and a maximum width of 480px. If your content is longer than 225px, allow the width to hug content until it reaches 480px.
- You can set a fixed width within the range, too. For example, to maintain visual consistency, give all Popovers in a series the same width based on the longest one.
- The title and body content wrap to the next line at maximum width to avoid truncation.

In this example, two Popovers explain different parts of Knowledge Share. Although the content length varies, the width is consistent based on the longer text in step one.

### Opening

- By default, a Popover opens after a user activates a Button or an Icon Button.
- When `initialOpen` is true, a Popover opens when its anchor is mounted.

A Popover that opens by itself receives the initial focus when a page first loads. You must indicate what the Popover points to in screen reader–only text. Otherwise, a person using a screen reader would not have the context to make sense of the paired relationship.

### Closing

If a Popover is initially open, turn off `dismissOnOutsideClick` to prevent a user from accidentally dismissing it, as they user would have no way to recover the lost information.

Otherwise, a Popover closes when a user:

- Activates the **Close** Icon Button in the upper-right corner of a Popover
- Activates a primary or tertiary Button in the Popover’s footer
- Activates another element or clicks outside of the Popover

#### Popovers on a Modal

When a Popover appears inside or on top of a Modal, be aware of how you let users close or back out of each layer.

The <kbd>Esc</kbd> key should close the Popover first and then the Modal.

### Multistep footer

Multiple Popovers connected in a series work best with passive, low-priority content—like definitions or brief feature highlights.

**A  maximum of three steps in a row is recommended.** Dense information is hard to remember and users tend to leave after just two steps. If you feel you need more than three steps to explain an interaction, it may indicate a larger design flaw. For instance, is the core user flow as efficient as it could be? 

When asking a user to take multiple actions:

- Only show the next piece of the tutorial after the currently requested action is complete.
- Use tools that already exist in components, such as helper text, to provide persistent instructions.
- Link to a support article for robust procedures.
- Display images or animations of the UI in a Modal to demonstrate unique interactions.

## Accessibility

### Focus order

A Popover should be accessible with the mouse cursor and keyboard.

When a Popover opens, the focus moves to the **Close** Icon Button. For people who use a screen reader, content is read in this order:

<ol>
  <li>Accessibility label (required)</li>
  <li>Close Icon Button (required)</li>
  <li>Illustration alt text</li>
  <li>Title</li>
  <li>Body (required)</li>
  <li>Footer content</li>
</ol>

### Labeling the Popover

An `accessibilityLabel` is required and must clearly communicate the purpose of the Popover. Make sure to factor in the Popover title when present and align it with the accessibility label in a way that will make sense when read aloud together.

#### If the title is not descriptive:

Add detail in the accessibility label when a title does not describe the context fully by itself. For example, if the title is just “New,” then you need to explain what specifically is new in the accessibility label.

#### If `initialOpen` is true:

Because a Popover that’s initially open receives the initial focus, a screen reader user needs to be told what it points to. Indicate the Popover’s anchor in the accessibility label to explain the paired relationship.

#### If there are multiple steps:

A screen reader user needs the context that they’re continuing on a tour of the same feature (and not getting disparate information about different features). You could provide content as follows:

- **Title**: Polish your drafts
- **`accessibilityLabel`**: Grammarly in Gmail tour, step 2 of 6, focus returns to Accept button.

In all six steps, it’s announced that the information is part of a single “Grammarly in Gmail tour” collection.

### Keyboard interaction

| Key                                  | Expected result                                                                                                                                             |
| ------------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------- |
| <kbd>Tab</kbd>                       | Moves focus through any focusable elements within the Popover. Once the end of the Popover is reached, focus loops back to the beginning of the Popover. |
| <kbd>Enter</kbd> or <kbd>Space</kbd> | Activates an action within the Popover or the Close Icon Button.                                                                                            |
| <kbd>Esc</kbd>                       | Closes the Popover. When a Popover is closed, focus returns to the element that activated it.                                                               |

### Body

Body content provides detail supporting the title or key concept.

Be brief and focused—use 140 characters or fewer. Using long sentences or mixing separate ideas can slow down a user with more information than they can absorb at one time.

Apply strategic writing tactics to break up dense information into smaller chunks. Tools to disperse content in chunks include:

- Bulleted lists
- Fragments or choppy sentences
- Multiple steps

### Footer

#### Primary action

Follow the Button documentation for actions in a Popover. [Learn how to write a Button](/components/button#writing).

When presenting information, offering both a **Close** Icon Button in the upper-right corner and a **Close** primary Button can make it easier for a user to dismiss the Popover.

#### Multistep

When presenting information in multiple steps:

- Use **Back** and **Next** for the tertiary and primary Buttons that navigate between steps.
- Omit the **Back** Button in the _first_ Popover of the series.
- Use **OK** as the primary Button in the _last_ Popover of the series.
